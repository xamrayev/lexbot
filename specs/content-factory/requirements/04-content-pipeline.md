# 04 — Requirements: Content Pipeline

**Модуль:** `PIPE` · **Статус:** Draft · **Зависит от:** 02, 03, 05, 06 · **Design:** `../design/pipeline.md`

---

## User Stories

- **US-PIPE-1**: Как владелец канала, я хочу, чтобы каждый день конвейер сам проходил
  путь от сбора информации до готового материала, чтобы моё участие сводилось к одобрению.
- **US-PIPE-2**: Как администратор платформы, я хочу конфигурировать каждый этап
  (модель, токены, температура, промпт, таймаут, ретраи, fallback) без изменения кода,
  чтобы оптимизировать стоимость и качество.
- **US-PIPE-3**: Как владелец, я хочу массовую генерацию (100 идей → 20 статей → 50
  твитов …), чтобы быстро наполнять несколько форматов.

## Канонический ежедневный конвейер

```
 1. Поиск нового контента          (ingestion, модуль SRC)
 2. Удаление дубликатов            (SRC)
 3. Суммаризация                   (cheap model)
 4. Извлечение фактов              (medium model)
 5. Проверка источников            (trust score, cross-check)
 6. Проверка актуальности          (freshness, уже публиковалось? — Memory)
 7. Генерация идей                 (medium model)
 8. План публикаций                (Content Planner)
 9. Написание статьи               (premium model + Style Profile)
10. Саморедактирование             (AI Editor)
11. SEO                            (cheap model)
12. Проверка качества              (Quality Score, модуль QS)
13. Отправка пользователю          (approval-очередь / уведомление)
14. Публикация                     (модуль PUB)
15. Сбор аналитики                 (модуль AN)
```

## Functional Requirements (EARS)

### Оркестрация

- **REQ-PIPE-001**: THE SYSTEM SHALL исполнять конвейер как durable workflow
  (Asynq на MVP, Temporal в Phase 2): каждый этап — отдельная задача с состоянием,
  ретраями и возможностью возобновления с места сбоя.
- **REQ-PIPE-002**: WHEN этап завершается с ошибкой после всех ретраев и fallback-модели,
  THE SYSTEM SHALL пометить запуск `failed` (или `degraded`, если этап необязательный),
  сохранить контекст ошибки и уведомить владельца канала.
- **REQ-PIPE-003**: THE SYSTEM SHALL запускать конвейер: по расписанию канала,
  вручную («сгенерировать сейчас») и по событию (например, `TrendDetected`).
- **REQ-PIPE-004**: THE SYSTEM SHALL публиковать события жизненного цикла запуска в
  Event Bus: `PipelineStarted`, `StageCompleted`, `ArticleGenerated`, `PipelineFailed`.
- **REQ-PIPE-005**: Каждый запуск SHALL иметь trace id, связывающий все этапы,
  LLM-вызовы и их стоимость (наследует Constitution X).

### Конфигурация этапов

- **REQ-PIPE-010**: THE SYSTEM SHALL хранить конфигурацию каждого этапа: модель
  (ссылка на AI Model из REQ-LLM-020), max tokens, temperature, системный промпт
  (ссылка на Prompt Library), таймаут, число ретраев, fallback-модель.
- **REQ-PIPE-011**: WHEN администратор изменяет конфигурацию этапа, THE SYSTEM SHALL
  применить её к следующим запускам без перекомпиляции и рестарта.
- **REQ-PIPE-012**: THE SYSTEM SHALL поддерживать Multi-Model Pipeline: разные этапы
  используют разные классы моделей (cheap/medium/premium) согласно конфигурации.
- **REQ-PIPE-013**: Промпты всех этапов SHALL храниться в Prompt Library
  (Generate Article, Rewrite, SEO, Summarize, Translate, Title, Fact Check, Review,
  Trend Analysis) с версионированием и возможностью редактирования в админке.

### Типы контента

- **REQ-PIPE-020**: THE SYSTEM SHALL поддерживать типы материалов: статья, новость,
  история, карусель, пост, thread, FAQ, разбор, обзор, сравнение, интервью, конспект,
  обучающий материал, инструкция, чеклист, рейтинг, дайджест, подборка, мнение,
  развенчание мифов, объяснение, кейс, email, landing, видео-сценарий, подкаст.
  Тип задаёт шаблон структуры и целевую длину.
- **REQ-PIPE-021**: Каждый тип SHALL описываться декларативно (шаблон структуры,
  ограничения длины, специфичный промпт), новые типы добавляются данными, не кодом.

### Массовая генерация

- **REQ-PIPE-030**: THE SYSTEM SHALL поддерживать batch-запуски: N идей → M материалов
  выбранных типов (например, 100 идей → 20 статей → 50 твитов → 30 постов → 10 email
  → 5 видео-сценариев).
- **REQ-PIPE-031**: WHEN batch-запуск превышает лимиты тарифа, THE SYSTEM SHALL
  остановить постановку новых задач на границе лимита и показать частичный результат.
- **REQ-PIPE-032**: Batch-запуски SHALL исполняться с ограничением параллелизма
  per-workspace, чтобы не деградировать интерактивные операции.

### Workflow Builder (Phase 2)

- **REQ-PIPE-040**: THE SYSTEM SHALL позволять собирать кастомный пайплайн из блоков
  (Source → Translate → Summarize → Generate → Rewrite → Publish → Analytics)
  в визуальном редакторе; результат сохраняется как декларативный DAG (JSON).
- **REQ-PIPE-041**: THE SYSTEM SHALL валидировать DAG при сохранении: ациклический,
  совместимость входов/выходов блоков, обязательный этап Quality Gate перед Publish.
- **REQ-PIPE-042**: Кастомные workflow SHALL быть экспортируемы в Marketplace
  (REQ-EXT-020).

## Non-Functional

- **NFR-PIPE-1**: Полный ежедневный запуск канала (≤ 50 новых элементов) — < 15 минут.
- **NFR-PIPE-2**: Сбой воркера не теряет запуск: состояние восстанавливается из хранилища задач.
- **NFR-PIPE-3**: Стоимость запуска считается и сохраняется в разрезе этапов (наследует Constitution VI).

## Acceptance Criteria

- [ ] AC-PIPE-1: e2e-тест — конвейер из 15 этапов проходит на фикстурных источниках и создаёт материал в approval-очереди. *(REQ-PIPE-001)*
- [ ] AC-PIPE-2: убийство воркера на этапе 9 → запуск возобновляется с этапа 9, этапы 1–8 не повторяются. *(REQ-PIPE-001, NFR-PIPE-2)*
- [ ] AC-PIPE-3: изменение модели этапа в админке влияет на следующий запуск без рестарта. *(REQ-PIPE-011)*
- [ ] AC-PIPE-4: недоступность premium-модели → этап Writing исполняется fallback-моделью, запуск помечен `degraded`. *(REQ-PIPE-002, REQ-PIPE-010)*
- [ ] AC-PIPE-5: DAG с циклом или без Quality Gate отклоняется валидатором. *(REQ-PIPE-041)*
- [ ] AC-PIPE-6: batch «10 идей → 3 статьи» останавливается на лимите тарифа с частичным результатом. *(REQ-PIPE-031)*
