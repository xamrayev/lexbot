# 08 — Requirements: Analytics & Feedback Loop

**Модуль:** `AN` · **Статус:** Draft · **Зависит от:** 05, 07 · **Design:** `../design/data-model.md`

---

## User Stories

- **US-AN-1**: Как владелец, я хочу видеть эффективность публикаций (просмотры, CTR,
  ER, подписки/отписки), чтобы понимать, что работает.
- **US-AN-2**: Как владелец, я хочу, чтобы ИИ сам менял стратегию на основе метрик,
  чтобы контент со временем становился лучше без моего вмешательства.
- **US-AN-3**: Как редактор, я хочу A/B-тесты заголовков и вступлений, чтобы выбирать
  лучшие варианты по данным, а не интуиции.

## Functional Requirements (EARS)

### Сбор метрик

- **REQ-AN-001**: WHEN материал опубликован, THE SYSTEM SHALL собирать по расписанию
  (1ч, 6ч, 24ч, 72ч, 7д после публикации) доступные метрики платформы: просмотры,
  CTR, комментарии, лайки, репосты, время чтения (если доступно), подписки, отписки.
- **REQ-AN-002**: Метрики SHALL храниться как time series с привязкой к материалу,
  типу контента, теме, заголовку, времени публикации и версии стиля.
- **REQ-AN-003**: WHEN платформа не отдаёт метрику, THE SYSTEM SHALL помечать её
  `unavailable`, не подставляя нулей в агрегаты.

### Дашборды

- **REQ-AN-010**: THE SYSTEM SHALL показывать по каналу: рост аудитории, CTR, ER,
  лучшие темы, лучшие часы публикации, лучшие заголовки, лучшие источники,
  конверсию и ROI (стоимость генерации против настроенной ценности результата).
- **REQ-AN-011**: THE SYSTEM SHALL показывать стоимость контента: по материалу,
  каналу, workspace, периоду (данные REQ-LLM-050).

### Feedback Loop

- **REQ-AN-020**: THE SYSTEM SHALL периодически (default — еженедельно) агрегировать
  метрики в выводы и записывать их в память канала (REQ-AGENT-020): успешные
  темы/типы/структуры/часы, неуспешные паттерны, темы с отписками.
- **REQ-AN-021**: WHEN выводы обновлены, THE SYSTEM SHALL использовать их в Planner
  (REQ-AGENT-012) и Writer (подбор структуры/тона) при следующих запусках.
- **REQ-AN-022**: Автоматические изменения стратегии SHALL быть видимы пользователю
  (лента «что ИИ изменил и почему») и отключаемы per-channel.
- **REQ-AN-023**: THE SYSTEM SHALL защищаться от малых выборок: выводы формируются
  только при достижении минимального объёма данных (настраиваемый порог), иначе
  помечаются `low_confidence` и не применяются автоматически.

### Эксперименты (A/B)

- **REQ-AN-030**: THE SYSTEM SHALL генерировать до N вариантов заголовка/вступления/CTA
  и, где платформа позволяет, публиковать варианты по сегментам; где нет — выбирать
  вариант по прогнозной оценке Reviewer и логировать для офлайн-сравнения.
- **REQ-AN-031**: WHEN эксперимент набрал целевой объём данных, THE SYSTEM SHALL
  определить победителя по заданной метрике и записать паттерн победителя в память.

## Acceptance Criteria

- [ ] AC-AN-1: после фикстурной публикации метрики собраны по всем контрольным точкам расписания. *(REQ-AN-001)*
- [ ] AC-AN-2: недоступная метрика не искажает агрегаты (`unavailable`, не 0). *(REQ-AN-003)*
- [ ] AC-AN-3: еженедельная агрегация создаёт записи памяти; Planner в тесте меняет рекомендации после их появления. *(REQ-AN-020..021)*
- [ ] AC-AN-4: при выборке ниже порога выводы имеют `low_confidence` и не применяются. *(REQ-AN-023)*
- [ ] AC-AN-5: A/B из 3 заголовков определяет победителя и сохраняет паттерн в память. *(REQ-AN-030..031)*
- [ ] AC-AN-6: [manual] дашборд отображает все метрики REQ-AN-010 на демо-данных.
