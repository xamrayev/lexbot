# 03 — Requirements: Sources & Ingestion

**Модуль:** `SRC` · **Статус:** Draft · **Зависит от:** 01 · **Design:** `../design/pipeline.md`, `../design/architecture.md`

---

## User Stories

- **US-SRC-1**: Как владелец канала, я хочу подключить свои источники (Telegram, RSS,
  сайты, YouTube, документы), чтобы ИИ ежедневно собирал из них новую информацию.
- **US-SRC-2**: Как редактор, я хочу задавать приоритет и уровень доверия источникам,
  чтобы влиять на отбор тем и проверку фактов.
- **US-SRC-3**: Как пользователь, я хочу собственную базу знаний (RAG) как основной
  источник истины, чтобы контент опирался на мои данные.

## Functional Requirements (EARS)

### Типы источников

- **REQ-SRC-001**: THE SYSTEM SHALL поддерживать типы источников (по фазам, см.
  `00-overview.md §6`): Telegram, RSS, Web (crawler), PDF, Word — *MVP*;
  YouTube (транскрипты), Twitter/X, Reddit, GitHub, Google Drive, Dropbox, Notion,
  Confluence, Email, Google Alerts, новостные API, произвольный API — *Phase 2+*;
  собственная база знаний (RAG) — *MVP*.
- **REQ-SRC-002**: Новые типы источников SHALL подключаться через Plugin SDK
  (интерфейс `SourceConnector`, см. Design) без изменения ядра.

### Конфигурация источника

- **REQ-SRC-010**: THE SYSTEM SHALL хранить для каждого источника: название, тип,
  приоритет (1–10), частоту обновления (cron/interval), фильтр (ключевые
  слова/регулярные выражения/язык), язык, уровень доверия (0–100).
- **REQ-SRC-011**: WHEN наступает время обновления источника, THE SYSTEM SHALL
  запустить fetch-задачу в очереди с ретраями (3 попытки, экспоненциальный backoff).
- **REQ-SRC-012**: WHEN источник недоступен 3 обновления подряд, THE SYSTEM SHALL
  пометить его `degraded`, уведомить владельца и продолжать попытки по расписанию.
- **REQ-SRC-013**: THE SYSTEM SHALL применять фильтр источника до сохранения элементов
  (нерелевантные элементы не тратят хранилище и токены).

### Ingestion & нормализация

- **REQ-SRC-020**: WHEN получен новый элемент, THE SYSTEM SHALL нормализовать его в
  единый формат `ContentItem`: id, source_id, url/ссылка на оригинал, заголовок, текст,
  язык, дата публикации, автор, media-ссылки, хеш содержимого.
- **REQ-SRC-021**: THE SYSTEM SHALL выполнять дедупликацию на двух уровнях:
  точная (хеш содержимого) и семантическая (косинусная близость embeddings выше
  порога 0.92 в пределах канала за скользящее окно 30 дней).
- **REQ-SRC-022**: WHEN элемент является дубликатом, THE SYSTEM SHALL связать его с
  оригиналом (для подсчёта «сколько источников подтверждают») и не создавать новую
  запись для пайплайна.
- **REQ-SRC-023**: THE SYSTEM SHALL сохранять исходный текст элемента неизменным
  (immutable) для последующей проверки фактов и плагиата (REQ-QS-010, REQ-QS-015).
- **REQ-SRC-024**: Файловые источники (PDF, Word) SHALL обрабатываться пайплайном
  извлечения текста с сохранением структуры (заголовки, списки, таблицы).

### Собственная база знаний (RAG)

- **REQ-SRC-030**: THE SYSTEM SHALL позволять каналу иметь базу знаний: загруженные
  документы, индексируемые в pgvector (+ полнотекстовый индекс).
- **REQ-SRC-031**: WHEN канал настроен в режиме «база знаний — основной источник
  истины», THE SYSTEM SHALL при проверке фактов (REQ-QS-010) считать её приоритетным
  подтверждением, а внешние источники — вторичными.
- **REQ-SRC-032**: THE SYSTEM SHALL выполнять hybrid search по базе знаний:
  BM25 + векторный поиск + RRF (переиспользуем подход LegalOS, см. `docs/legalos/architecture.md`).

## Non-Functional

- **NFR-SRC-1**: Обновление одного RSS-источника (100 элементов) — < 30 сек.
- **NFR-SRC-2**: Fetch-задачи разных каналов не блокируют друг друга (per-channel очереди/семафоры).
- **NFR-SRC-3**: Rate limits внешних API соблюдаются per-provider (token bucket), ключи — в Secrets Manager.

## Acceptance Criteria

- [ ] AC-SRC-1: RSS-источник с фильтром «AI» сохраняет только элементы, содержащие ключевое слово. *(REQ-SRC-013)*
- [ ] AC-SRC-2: два одинаковых поста из разных источников создают один `ContentItem` с двумя ссылками-подтверждениями. *(REQ-SRC-021..022)*
- [ ] AC-SRC-3: PDF со списками и таблицами извлекается со структурой. *(REQ-SRC-024)*
- [ ] AC-SRC-4: тест — недоступный источник помечается `degraded` после 3 неудач, уведомление отправлено. *(REQ-SRC-012)*
- [ ] AC-SRC-5: hybrid search по базе знаний возвращает релевантный документ для эталонных запросов (eval-набор). *(REQ-SRC-032)*
