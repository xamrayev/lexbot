# 01 — Requirements: Workspace, Channels, Team

**Модуль:** `WS` · **Статус:** Draft · **Зависит от:** — · **Design:** `../design/architecture.md`, `../design/data-model.md`

---

## User Stories

- **US-WS-1**: Как владелец, я хочу создать workspace и в нём несколько независимых каналов,
  чтобы управлять контентом нескольких проектов из одного аккаунта.
- **US-WS-2**: Как владелец, я хочу приглашать команду с ролями, чтобы распределить работу
  редакции и контролировать доступ.
- **US-WS-3**: Как пользователь нескольких компаний, я хочу переключаться между workspace,
  чтобы вести несколько бизнесов (Multi Workspace).

## Functional Requirements (EARS)

### Workspace

- **REQ-WS-001**: WHEN пользователь регистрируется, THE SYSTEM SHALL создать для него
  workspace по умолчанию и назначить роль `owner`.
- **REQ-WS-002**: THE SYSTEM SHALL позволять одному пользователю состоять в нескольких
  workspace и переключаться между ними без повторной аутентификации.
- **REQ-WS-003**: WHEN пользователь создаёт workspace, THE SYSTEM SHALL проверить лимит
  тарифа на количество workspace и отклонить создание с ошибкой `limit_exceeded`
  при превышении.
- **REQ-WS-004**: THE SYSTEM SHALL изолировать данные workspace: любой запрос без
  подтверждённого членства в workspace SHALL возвращать `404` (не `403`, чтобы не
  раскрывать существование ресурса).

### Channels

- **REQ-WS-010**: WHEN участник с ролью `owner`/`admin` создаёт канал, THE SYSTEM SHALL
  инициализировать для него независимые: Brand, Style Profile, Sources, Schedule,
  Analytics, Memory и конфигурацию AI Agent.
- **REQ-WS-011**: THE SYSTEM SHALL гарантировать, что данные канала (память, стиль,
  источники, аналитика) недоступны из контекста других каналов, включая LLM-контекст
  агентов (изоляция по `channel_id`).
- **REQ-WS-012**: WHEN канал удаляется, THE SYSTEM SHALL выполнить soft-delete с периодом
  восстановления 30 дней, а затем безвозвратно удалить все связанные данные, включая
  embeddings и память.
- **REQ-WS-013**: WHEN создание канала превышает лимит тарифа, THE SYSTEM SHALL отклонить
  операцию до вызова каких-либо платных ресурсов.
- **REQ-WS-014**: THE SYSTEM SHALL хранить для канала режим публикации:
  `auto` | `approval` | `co-edit`; значение по умолчанию — `approval`.

### Team & Roles

- **REQ-WS-020**: THE SYSTEM SHALL поддерживать роли: `owner`, `admin`, `editor`
  (Редактор), `author` (Автор), `seo`, `viewer` (Наблюдатель).
- **REQ-WS-021**: WHEN участник приглашается по email, THE SYSTEM SHALL отправить
  приглашение со сроком действия 7 дней и активировать членство только после принятия.
- **REQ-WS-022**: THE SYSTEM SHALL применять матрицу прав (см. Design) на уровне
  middleware; UI-скрытие элементов не считается контролем доступа.
- **REQ-WS-023**: WHEN участник оставляет комментарий к материалу, THE SYSTEM SHALL
  сохранить его с привязкой к версии материала и уведомить причастных.
- **REQ-WS-024**: THE SYSTEM SHALL вести историю изменений (audit log) всех действий над
  каналами, материалами и настройками: кто, что, когда, до/после.
- **REQ-WS-025**: WHEN материал требует одобрения (режим `approval`), THE SYSTEM SHALL
  разрешать публикацию только участникам с ролью `owner`/`admin`/`editor`.

## Acceptance Criteria

- [ ] AC-WS-1: интеграционный тест — пользователь A не может прочитать канал workspace B (404). *(REQ-WS-004)*
- [ ] AC-WS-2: тест — создание канала сверх лимита Free-тарифа возвращает `limit_exceeded`. *(REQ-WS-013)*
- [ ] AC-WS-3: тест — `author` не может опубликовать материал в режиме `approval`. *(REQ-WS-025)*
- [ ] AC-WS-4: тест — удалённый канал восстановим в течение 30 дней; после — данные и embeddings отсутствуют. *(REQ-WS-012)*
- [ ] AC-WS-5: audit log фиксирует изменение настроек канала с diff до/после. *(REQ-WS-024)*
