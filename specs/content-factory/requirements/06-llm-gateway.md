# 06 — Requirements: LLM Gateway & Providers

**Модуль:** `LLM` · **Статус:** Draft · **Зависит от:** — · **Design:** `../design/llm-gateway.md`

---

## User Stories

- **US-LLM-1**: Как администратор платформы, я хочу добавлять провайдеров и модели
  через админку (base URL, ключ, цены), чтобы менять модельный парк без релиза.
- **US-LLM-2**: Как пользователь, я хочу выбирать «Высокое качество / Быстро /
  Экономично», не зная деталей провайдеров.
- **US-LLM-3**: Как администратор, я хочу автоматический failover при недоступности
  модели, чтобы пайплайны не падали.

## Functional Requirements (EARS)

### Gateway (единая точка доступа)

- **REQ-LLM-001**: THE SYSTEM SHALL предоставлять единственную точку доступа к LLM —
  LLM Gateway; прямые вызовы провайдеров из бизнес-логики SHALL быть запрещены
  (контроль — линтером/arch-тестом на импорты).
- **REQ-LLM-002**: THE SYSTEM SHALL определять интерфейс провайдера с операциями:
  `Chat` (включая streaming и tool use), `Embedding`, `Image`, `SpeechToText`,
  `TextToSpeech`, `Rerank`, `Moderation`. Провайдер MAY реализовывать подмножество,
  декларируя capabilities.
- **REQ-LLM-003**: THE SYSTEM SHALL поддерживать провайдеров: OpenAI, OpenRouter,
  vLLM, Ollama, LM Studio, LiteLLM, DeepSeek, Groq, Together AI, Fireworks, Cerebras,
  Mistral — через OpenAI-совместимый адаптер (base URL + key); Anthropic и
  Google Gemini — через нативные адаптеры; Azure OpenAI — через адаптер;
  локальные модели — через Ollama/vLLM.
- **REQ-LLM-004**: WHEN добавляется новый провайдер, THE SYSTEM SHALL требовать только
  реализацию интерфейса (или конфигурацию OpenAI-совместимого endpoint) — без
  изменения ядра (Constitution II).
- **REQ-LLM-005**: THE SYSTEM SHALL логировать каждый вызов: trace id, workspace,
  канал, этап, модель, провайдер, токены in/out, латентность, стоимость, статус.
  Содержимое промптов SHALL логироваться отдельно с настраиваемой ретенцией и
  редактированием секретов.

### Модели и классы качества

- **REQ-LLM-020**: Администратор SHALL создавать «AI Model» как связку:
  отображаемое имя, провайдер, model id, тип (chat/embedding/vision/image/audio),
  цены input/output, лимиты, флаг активности.
- **REQ-LLM-021**: THE SYSTEM SHALL сопоставлять модели классам качества
  (`premium` / `fast` / `economy`); пользователь выбирает класс, разрешение класса
  в конкретную модель выполняет Gateway согласно тарифу и конфигурации.
- **REQ-LLM-022**: WHEN модель выключается администратором, THE SYSTEM SHALL
  немедленно исключить её из маршрутизации; активные запросы завершаются.
- **REQ-LLM-023**: API-ключи провайдеров SHALL храниться в Secrets Manager
  (шифрование at rest, доступ только у Gateway); в API-ответах и логах ключи
  SHALL маскироваться всегда.

### AI Router & Failover

- **REQ-LLM-030**: WHEN модель возвращает ошибку доступности/перегрузки или превышает
  таймаут, THE SYSTEM SHALL выполнить failover по настроенной цепочке (пример:
  GPT-5 → GPT-5 Mini → DeepSeek → Qwen) с сохранением семантики запроса.
- **REQ-LLM-031**: THE SYSTEM SHALL применять circuit breaker per-model: после N
  последовательных сбоев модель временно выводится из ротации с постепенным
  возвратом (half-open).
- **REQ-LLM-032**: Каждый failover SHALL логироваться и учитываться в метриках;
  результат, полученный fallback-моделью, помечается в trace.
- **REQ-LLM-033**: THE SYSTEM SHALL соблюдать rate limits провайдеров (token bucket
  per provider key) и ставить запросы в очередь вместо мгновенного отказа, пока не
  превышен таймаут этапа.

### Prompt Library

- **REQ-LLM-040**: THE SYSTEM SHALL хранить все промпты в Prompt Library с полями:
  ключ, версия, текст, переменные, описание; редактирование — без перекомпиляции.
- **REQ-LLM-041**: WHEN промпт обновляется, THE SYSTEM SHALL сохранять предыдущие
  версии и позволять откат; активная версия выбирается явно.
- **REQ-LLM-042**: Рендеринг промпта SHALL валидировать, что все обязательные
  переменные заполнены; незаполненная переменная — ошибка этапа до вызова модели.

### Cost Control

- **REQ-LLM-050**: THE SYSTEM SHALL рассчитывать стоимость каждой генерации из цен
  модели и фактических токенов и агрегировать её по: вызов → этап → запуск → канал →
  workspace → период.
- **REQ-LLM-051**: WHEN лимит тарифа (сообщения/токены/бюджет) исчерпан, THE SYSTEM
  SHALL отклонить вызов ДО обращения к провайдеру с ошибкой `quota_exceeded`
  (Constitution VI.4).
- **REQ-LLM-052**: THE SYSTEM SHALL поддерживатьмесячный бюджет workspace c мягким
  (уведомление на 80%) и жёстким (стоп) порогами.

## Non-Functional

- **NFR-LLM-1**: Overhead Gateway (без времени провайдера) — p99 < 50 мс.
- **NFR-LLM-2**: Gateway горизонтально масштабируем и stateless (состояние — в Redis/PG).
- **NFR-LLM-3**: Streaming-ответы проксируются без буферизации целиком.

## Acceptance Criteria

- [ ] AC-LLM-1: arch-тест — модули вне `internal/llm` не импортируют SDK провайдеров. *(REQ-LLM-001)*
- [ ] AC-LLM-2: добавление OpenAI-совместимого провайдера через админку (base URL + key) делает его модели доступными без релиза. *(REQ-LLM-004, REQ-LLM-020)*
- [ ] AC-LLM-3: тест — отказ primary-модели ведёт к ответу от fallback; trace содержит отметку failover. *(REQ-LLM-030, REQ-LLM-032)*
- [ ] AC-LLM-4: тест — исчерпанный лимит блокирует вызов до обращения к провайдеру. *(REQ-LLM-051)*
- [ ] AC-LLM-5: стоимость запуска пайплайна равна сумме стоимостей LLM-вызовов его этапов (сверка в тесте). *(REQ-LLM-050)*
- [ ] AC-LLM-6: незаполненная переменная промпта даёт ошибку до вызова модели. *(REQ-LLM-042)*
