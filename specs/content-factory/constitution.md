# Content Factory (ContentOS) — Constitution

> Основополагающий документ проекта. Все требования (`requirements/`), проектные решения (`design/`)
> и задачи (`tasks/`) обязаны соответствовать этим принципам. Изменение конституции — отдельный PR
> с обоснованием и мажорным версионированием документа.

**Версия:** 1.0.0 · **Статус:** Draft · **Дата:** 2026-07-18

---

## Видение

ContentOS — это не генератор статей, а **операционная система для управления контентом**.
Пользователь один раз настраивает канал, после чего ИИ самостоятельно ищет материалы,
анализирует их, пишет контент, планирует публикации и постоянно обучается на реакции аудитории.
Пользователь становится главным редактором, ИИ — полноценной редакцией.

## Статьи конституции

### I. Spec-Driven Development (NON-NEGOTIABLE)

1. Ни одна фича не реализуется без утверждённой спецификации в `specs/content-factory/requirements/`.
2. Порядок работы строго: **Requirement → Design → Tasks → Implementation → Verification**.
3. Каждое требование записывается в EARS-нотации (`WHEN … THE SYSTEM SHALL …`) и имеет
   уникальный ID вида `REQ-<MODULE>-<NNN>` (пример: `REQ-PIPE-003`).
4. Каждая задача в `tasks/` ссылается на ID требований, которые она реализует.
5. Каждый acceptance criterion должен быть проверяем автоматическим тестом либо
   явно помечен как `[manual]`.

### II. Provider Abstraction (NON-NEGOTIABLE)

1. Бизнес-логика **никогда** не обращается к API конкретного LLM-провайдера напрямую.
   Единственная точка доступа — LLM Gateway (интерфейс `Provider`).
2. Любой новый провайдер подключается реализацией интерфейса, без изменения ядра.
3. Пользователь оперирует классами качества («Высокое качество» / «Быстро» / «Экономично»)
   и никогда не видит API конкретного провайдера.
4. Смена модели или провайдера не требует изменения бизнес-логики.

### III. Channel Isolation

1. Каждый канал полностью независим: собственный стиль, источники, расписание,
   аналитика, память и AI-агент.
2. Данные каналов не пересекаются; память одного канала недоступна агентам другого,
   кроме явного экспорта через Marketplace.
3. Multi-tenancy: Workspace → Channels → изоляция на уровне строк БД (`workspace_id`,
   `channel_id` в каждой доменной таблице) + проверка в middleware.

### IV. Modular Monolith First

1. Старт — единое Go-приложение с жёсткими границами модулей (internal packages,
   общение только через интерфейсы и Event Bus).
2. Модули-кандидаты на выделение в сервисы: AI Service, Scheduler, Analytics, Billing,
   Search, Notification. Их публичные контракты проектируются как будущие сетевые API.
3. Прямые импорты между доменными модулями запрещены — только контракты (`internal/contract`)
   и события.

### V. Event-Driven Core

1. Все ключевые доменные действия публикуются как события в Event Bus
   (`ArticleGenerated`, `PostPublished`, `SourceUpdated`, `FeedbackCollected` и т.д.).
2. Новые возможности (аналитика, уведомления, интеграции, webhooks) подключаются
   как подписчики, не меняя издателей.
3. События имеют версионируемые схемы; обратная совместимость обязательна в пределах
   мажорной версии.

### VI. Cost Control by Design

1. Каждый LLM-вызов логируется с моделью, токенами и стоимостью.
2. Каждый этап пайплайна конфигурируется: модель, max tokens, температура, системный
   промпт, таймаут, retries, fallback-модель — без перекомпиляции.
3. Дорогие модели используются только там, где это оправдано (Multi-Model Pipeline);
   дефолтная стратегия — «самая дешёвая модель, проходящая порог качества».
4. Лимиты подписки применяются **до** вызова провайдера, а не после.

### VII. Human-in-the-Loop

1. Режим публикации настраивается на уровне канала: `auto` / `approval` / `co-edit`.
2. По умолчанию — `approval`: ничто не публикуется без одобрения человека.
3. Все версии материалов сохраняются; любой откат — в один клик; изменения ИИ
   отличимы от изменений человека (diff + авторство).

### VIII. Trust & Safety

1. Каждое фактическое утверждение в сгенерированном материале имеет ссылку на
   первоисточник и уровень доверия.
2. Перед публикацией — проверка на плагиат относительно исходных материалов;
   слишком похожие фрагменты переписываются автоматически.
3. Канальные правила (запрещённые слова, обязательные дисклеймеры, фирменная
   терминология) применяются как жёсткий пост-фильтр, а не только через промпт.
4. Секреты (API-ключи провайдеров, токены публикации) хранятся только в
   Secrets Manager, шифруются at rest, никогда не логируются и не попадают в LLM-контекст.

### IX. Quality Gates

1. Каждый материал перед публикацией получает Quality Score (0–100) по критериям:
   достоверность, читабельность, SEO, соответствие стилю, вероятность вовлечения.
2. Материалы ниже порогового значения канала не публикуются автоматически ни в одном режиме.
3. Пайплайн генерации обязан включать этапы саморедактирования и проверки качества.

### X. Observability

1. Каждый запуск пайплайна трассируется end-to-end (trace id через все этапы и LLM-вызовы).
2. Метрики (Prometheus) обязательны для: латентности этапов, стоимости, ошибок провайдеров,
   очередей, публикаций.
3. Ошибка любого этапа не теряется: retry по политике → fallback-модель → пометка запуска
   как degraded/failed с сохранением контекста для разбора.

## Глоссарий

| Термин | Определение |
|---|---|
| **Workspace** | Верхнеуровневый tenant (компания/команда), содержит каналы и команду |
| **Channel** | Независимая единица вещания со своим стилем, источниками, памятью и агентом |
| **Source** | Подключённый источник контента (Telegram, RSS, YouTube, PDF …) |
| **Pipeline Run** | Один запуск ежедневного (или ручного) конвейера генерации контента |
| **Style Embedding** | Векторное + структурированное представление авторского стиля канала |
| **AI Agent** | Оркестрируемая последовательность специализированных под-агентов |
| **LLM Gateway** | Единая точка доступа ко всем LLM-провайдерам |
| **Quality Score** | Интегральная оценка материала 0–100 перед публикацией |
| **Channel Memory** | Долговременная память канала: публикации, реакции, запреты, предпочтения |

## Структура спецификаций

```
specs/content-factory/
├── constitution.md            ← этот документ
├── 00-overview.md             ← видение, скоуп, персоны, домены
├── requirements/              ← EARS-требования по модулям (REQ-*)
├── design/                    ← техническое проектирование (DES-*)
└── tasks/                     ← фазы реализации, задачи со ссылками на REQ-*
```

## Правила изменения

- Новое требование → PR c изменением файла в `requirements/` + обновление затронутых design/tasks.
- Противоречие требования конституции → сначала правится конституция (или требование отклоняется).
- Каждый REQ-ID неизменяем; устаревшие требования помечаются `Deprecated`, но не удаляются.
